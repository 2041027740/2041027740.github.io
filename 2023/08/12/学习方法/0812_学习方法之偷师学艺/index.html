学习方法之偷师学艺

发表于 2023-08-12 更新于 2023-09-08 分类于 学习方法 阅读次数：
大佬的学习方法，学习思路，答题思路，学到就是我的了（未完待续）。

第一条：

回答问题的时候，首先先总结一条主线，按照这条主线去捋下来，这样让自己有一个清晰的思路，不会漏掉重要的知识点

举例：方法重载和方法重写的区别

在回答这个问题的时候我们先梳理一条主线——>按照我们写代码的顺序去 记忆

方法重载比较简单，先说方法重载

———

方法重载是同一个类中，多个同名的方法构成了重载，而方法重写，是发生在父子类继承关系之中，子类重写父类的方法。

构成方法重载需要满足的条件是

  • 权限修饰符，不做要求；方法名必须相同，形参列表必须不同（数据类型不同/个数不同/位置不同），返回值类型也没有做任何要求

构成方法重写需要满足的条件是

（子类重写父类方法）

  1. 访问权限修饰符不能比父类更严格
  2. 方法名，形参列表必须相同(就是方法的签名必须相同)，
  3. 子类不能比父类抛出更多的编译时异常
  4. 返回值类型如果父类返回父类类型，子类可以返回父类类型也可以返回子类类型。

第二条

学习包括记忆的时候，理解去记忆，梳理他整个的执行流程，执行的过程发生了什么事情，按照流程来和知识点关联起来，这样知识可以连接起来

举例：JVM运行时数据区由哪几部分构成，并分别简单介绍以下它们

  1. 一个程序运行(程序计数器工作)
  2. 类加载（将类的信息、静态成员、常量放在方法区）
  3. 创建对象（对象放在堆上、引用放在jvm栈上）
  4. 调用方法（方法栈帧进JVM栈）
  5. 调用本地方法（本地方法栈帧）

第一条、第二条的关联性：

梳理一条主线

  • 这条主线，可以按照我们写代码的顺序
  • 可以是代码执行的顺序，代码执行过程发生的情况，通过执行流程串联知识点
  • 注意：一定要将知识串联起来，死记硬背会造成知识体系紊乱、回答问题逻辑不清晰、遗漏重点等。

第三条

在回答两者之间的区别的时候，有些是可以通过实际业务中的选择，来将知识点穿起来，为什么选择这个，选这个的好处在哪里，缺点呢，不选另一个的原因，二者最大的区别在哪里。

选择这种方式回答

  1. 可以让你的逻辑更清晰，记忆的负担减小
  2. 可以带入自己的思考，和一些例子，用自己的话去回答这个知识点，这就预防别人以为你在背面经。

注意：可以对某方面深入钻研一下，将面试官带入到你熟悉擅长的领域，自己不熟悉的名词注意避开。

第四条

对学过的知识进行总结，及时回顾，当学完一个框架之后，要明白它的底层是用到了那些技术（对那些技术的一些封装），这样可以让自己把知识串联起来，同样通过回顾和总结，可以将这个框架理解的更加透彻

举例：servlet :

servlet是什么呢？

servlet（服务器小程序）它是将网络编程、多线程、http协议进行了封装

如果我们不使用servlet的话

  • 建立连接
    首先客户端需要创建套接字Socke，来与服务器建立连接（包括服务器的ip地址和端口号），服务端需要创建一个套接字（Socket）来监听端口。
  • 发送请求报文、解析请求报文
    连接建立成功，客户端发送请求报文，请求报文通过socket套接字，发送给服务器，服务器对这个请求报文进行解析，（请求路径、请求头请求方法、请求头、请求体）
  • 处理业务逻辑（数据库查询、读写）
    服务器根据这个请求的信息执行相应的业务逻辑，可能涉及数据库的查询、读写之类的
  • 返回响应报文，解析响应报文
    服务器处理完请求，构建响应报文通过socket发送给客户端，客户端需要解析响应报文，提取出服务器返回的数据。
  • 关闭连接，释放资源

这个过程需要手动编写大量的网络编程代码，包括 Socket 连接的创建、HTTP 报文的构建和解析、多线程处理等

要使用 Servlet 开发 Web 应用程序

  1. 创建 Servlet 类：首先，你需要创建一个 Java 类，该类将扩展 Servlet 接口或继承 HttpServlet 类。这个类将作为你的 Servlet 组件，用于处理 HTTP 请求和生成 HTTP 响应。

  2. 实现 Servlet 方法：在你的 Servlet 类中，你需要实现至少一个 Servlet 方法。最常见的方法是 doGet() 和 doPost()，它们用于处理 GET 和 POST 请求。你可以根据需要实现其他方法，如 doPut()、doDelete() 等。

  3. 配置 Servlet：你需要在 web.xml 文件中或使用注解为 Servlet 配置 URL 映射和其他初始化参数。在 web.xml 中，你需要添加 <servlet> 和 <servlet-mapping> 元素来定义 Servlet 和它的映射路径。或者使用注解配置（现在我们一般使用注解来配置）

    @WebServlet("/myservlet")

  4. 编写业务逻辑：在 Servlet 类中，你需要编写处理请求的业务逻辑。这可能涉及从请求中获取参数、执行数据库操作、生成响应等。

两者对别

不使用 Servlet：

  1. 原始网络编程： 如果不使用 Servlet，需要使用原始的套接字编程来处理 HTTP 请求和响应。这意味着需要自己编写代码来解析 HTTP 请求报文和生成响应报文，很复杂和繁琐。
  2. 多线程管理： 需要手动管理多个线程来处理多个并发请求。这包括为每个请求创建一个新线程，确保线程安全，以及处理线程的生命周期。
  3. HTTP 协议处理： 需要编写代码来解析 HTTP 协议的各个部分，包括请求方法（GET、POST 等）、请求头、响应状态码等。
  4. 路由和请求分派： 需要编写代码来实现请求的路由和分派，以确定哪个处理程序应该处理特定的请求。

使用 Servlet：

  1. 高级封装： Servlet 提供了高级封装，简化了 Web 开发。只需要编写 Java 类，实现 Servlet 接口或继承 HttpServlet 类，然后重写相应的方法来处理请求。
  2. 自动多线程处理： Servlet 容器（如Tomcat）自动管理多线程，每个请求都由一个独立的线程处理，无需手动处理多线程问题。
  3. HTTP 协议集成： Servlet 容器处理了大部分 HTTP 协议的细节，包括请求解析、响应生成以及会话管理。你可以专注于业务逻辑而不用处理这些底层细节。
  4. 内置路由和分派： Servlet 容器提供了内置的路由和请求分派机制，可以根据 URL 映射到适当的 Servlet，无需自己编写路由逻辑。
