SpringCloud

架构理论

单体架构

单体架构的特征

  • 单体架构是一种传统的应用程序架构，它将整个应用程序作为一个单一的、单独的单元来构建和部署。

单体架构的优势

  • 应用的开发很简单。
  • 易于对应用程序进行大规模的更改。
  • 测试相对直观简单。
  • 部署简单明了。
  • 横向扩容不费吹灰之力。

单体架构的弊端

  • 代码复杂，严重耦合，难以维护。
  • 从代码提交到实际部署的周期很长。
  • 扩展性受限。
  • 开发慢，启动慢，严重影响开发效率。
  • 交付可靠的单体应用困难。

微服务架构

什么是微服务架构

  • 微服务架构是一种应用程序架构风格，将应用程序拆分为一组小型、独立的服务，每个服务都有自己的数据存储、业务逻辑和用户界面，并可以独立部署和扩展。

什么是微服务

  • 每个微服务都是一个独立的应用程序单元，通常根据单一职责原则实现一组相关功能。

微服务架构的优势

  • 每个服务都相对较小，容易维护。
  • 快速的持续交付和持续部署。
  • 应用扩展灵活。
  • 更好的容错性。

微服务架构的弊端

  • 分布式系统可能复杂难以管理。
  • 分布式部署追踪问题难。
  • 分布式事务比较难处理。
  • 服务数量增加，管理复杂性增加。

微服务的实现

微服务有哪两种常见的服务划分方式

  • 横向划分：按照业务划分。
  • 纵向划分：按照层次划分（表现层、业务层、持久层）。

实现微服务

自己实现遇到的问题

  • 服务间的调用问题，包括通信方式和协议。
  • 各个服务的管理问题。

微服务框架解决以上问题

  • Dubbo：用于服务之间的通信和调用。
  • Spring Cloud：提供了一整套微服务架构的解决方案。

Spring Cloud基于Spring Boot提出

  • 服务注册与自动发现：Eureka、Nacos。
  • 面向接口的服务调用：Feign。
  • 服务调用的负载均衡：Ribbon。
  • 服务网关：Zuul、Gateway。
  • 服务熔断：Hystrix。

服务间的调用问题

服务调用的本质

  • 服务调用本质是将方法调用转换为HTTP请求。

服务调用的目标

  • 服务调用的目标是通过一次HTTP请求获取响应。

服务调用基于的协议

  • 服务调用基于HTTP协议。

实现最简单的服务调用

  • 使用RestTemplate来发送HTTP请求。

    通过一个RestTempate对象，发起POST,GET,DELETE,PUT,PATCH等不同种类的Http请求

    eg：发送GET请求

    先准备好一个用于发起Http请求的RestTemplate对象

服务的动态发现问题

如何让服务消费者动态发现服务提供者的信息

  • 使用服务注册中心来实现服务的注册与自动发现。

服务注册中心

  • 服务注册中心是用于管理服务实例的中心化组件。
Eureka
  • Eureka是一种服务注册中心，包括Eureka Server和Eureka Client两部分。
Eureka Server
  • Eureka Server用于实现服务注册中心的功能。
Eureka Client
  • Eureka Client用于实现服务提供者和服务消费者的注册和发现。
Nacos
  • Nacos是另一种服务注册中心，包括Nacos Server和Nacos Client两部分。
Nacos Server
  • Nacos Server用于实现服务注册中心的功能。
Nacos Client
  • Nacos Client用于实现服务提供者和服务消费者的注册和发现。

服务注册中心的实现

  • Eureka Server和Nacos Server都是服务注册中心的实现。

服务调用的负载均衡问题

服务消费者如何选择一个服务提供者实例发起调用

  • 使用负载均衡器来选择一个服务提供者实例。

Ribbon

  • Ribbon是一个负载均衡器，用于在多个服务提供者实例中选择一个进行调用。
Ribbon的基本使用
  • 使用Ribbon的@LoadBalance注解来标识RestTemplate，以实现负载均衡。
Ribbon支持的负载均衡策略
  • Ribbon支持多种负载均衡策略，如随机策略、轮询策略、重试策略等。
Ribbon指定调用服务时的负载均衡策略
  • 可以通过配置文件或配置类来指定调用服务时使用的负载均衡策略。
自定义负载均衡策略
  • 可以自定义负载均衡策略，并通过配置类注册。

面向接口的服务调用问题

能否以接口方法调用的形式完成服务调用

  • 是的，可以使用OpenFeign实现面向接口的服务调用。

OpenFeign

  • OpenFeign是一种面向接口的服务调用工具，它可以将接口方法调用转化为HTTP请求。
实现面向接口服务调用的底层原理
  • OpenFeign底层通过动态代理将接口方法转化为HTTP请求。
使用OpenFeign

  • 可以定义接口方法，参数和返回值为JDK中的数据类型，然后使用OpenFeign来调用服务。

  • 如果参数或返回值为自定义数据类型，也可以使用OpenFeign来调用服务。

打印OpenFeign请求日志
  • 可以配置OpenFeign来打印请求日志，便于调试和监控。
设置服务调用的超时时间
  • 可以配置OpenFeign来设置服务调用的超时时间，以防止调用超时。

服务的配置管理问题

如何简单、高效地完成对多个服务实例配置的管理

  • 使用配置中心来统一管理多个服务实例的配置。

配置中心

  • 配置中心是用于集中管理配置信息的组件。
Nacos配置中心的概念
  • Nacos配置中心包括profile配置和shared配置。
Profile配置

  • Profile配置允许不同的服务实例根据其profile属性获取不同的配置。

  • 配置集的计算公式定义了不同profile属性的配置。

Shared配置

  • Shared配置允许多个服务实例共享同一份配置。

  • 可以定义多个shared配置，并让多个服务实例引用相同的shared配置。

配置的访问方式
  • 配置可以通过访问配置中心获取，通常配置中心的地址放在bootstrap.yml配置文件中。
配置的隔离
  • 可以通过配置集、配置分组、命名空间等方式实现配置的隔离。
配置的动态刷新
  • 可以实现配置的动态刷新，使配置变更后不需要重启服务即可生效。
Profile配置、Shared配置和本地配置的优先级
  • 配置的优先级由profile配置、shared配置和本地配置共同决定。
数据源的切换
  • Nacos配置可以存储在不同的数据源中，例如MySQL，可以切换数据源来存储配置。
