Redis介绍

引入背景

web1.0(静态页面，没有互动)

web2.0(内容互动，数据爆炸)

传统关系型数据库

  • 数据的高并发——》实时获取——》上万读写，io承受不住，难以承受高并发
  • SNS(社交网站)
  • 特性无用武之地
    • 不严格要求数据库的事务，对读一致性要求低
    • 忌讳多个大表的关联查询，往往只做单表查询（自己组装数据）

Redis是一个NoSql数据库

  • 纯内存数据库（访问快）

  • key-value键值对存储数据

    • 设置键值对：set key value
    • 获取键对应的值：get key

  • 持久化（保存日志）——》目的：恢复数据

  • 支持网络：（客户端通过网络连接到服务端）CS架构

    • redis-server（启动）

    • redis-cli(我们用cli访问)

      • 默认本地的6379端口

持久化配置

内存中数据保存的磁盘文件中

RDB(快照技术)

快照：内存技术，保存内存当时的状态

  • 快照速度很快
  • 快照保存数据的状态，占用磁盘空间比较小
  • 快照会保存完整的数据

配置

  • dir ./ #快照保存的位置(默认当前路径)

  • dbfilename xxx.rdb #名字

快照触发策略（默认的三个时机，相互独立）
save <seconds变化的时间间隔> <changes键值对个数的变化>

  • save 900 1
  • save 600 10
  • save 300 10000
  • 正常终止Redis会触发RDB
  1. RDB保存的数据占用较小的磁盘空间
  2. RDB可能丢数据，上一次持久化之后所有写入的数据
  3. 数据速度快，还原也快

AOF持久

Append only file (不存数据，只存命令)

AOF持久化是将对数据的操作命令追加到日志文件中，具有以下特点：

  • 不会一直膨胀，会定期进行精简（监听）。
  • 可以记录数据的修改操作。

AOF的配置包括：

  • appendonly：是否启用AOF持久化，默认为关闭。
  • appendfilename：AOF文件的名字，默认为appendonly.aof。
  • appendfsync：数据同步策略保存策略可以设置为：
  • appendfsync everysec：每秒同步一次。
  • appendfsync always：每次写操作都同步。
  • appendfsync no：依赖于操作系统策略。

AOF可以保证不丢失数据，但生成的文件会越来越大，恢复数据时需要执行文件中的所有命令，速度较慢。

优先级

当同时启用RDB和AOF时，Redis在恢复数据时会优先使用AOF生成的文件。

常用五种数据结构

Redis支持多种数据结构，包括：

  1. String：存储键值对，支持增减操作。
  2. List：有序、可重复的数据结构。
  3. Set：无序、不可重复的数据结构。
  4. Hash：键值对集合。
  5. Zset（有序集合）：用于排行榜等应用。

string

内存淘汰策略

Redis使用不同的策略来淘汰数据以释放内存。常见的策略包括：

  • no-eviction：禁止淘汰数据，如果内存不足则写入操作报错。
  • volatile-lru：最近最少使用的数据被淘汰。
  • lfu：一段时间内使用最少的数据被淘汰。
  • random：随机淘汰数据。
  • ttl：淘汰将要过期的数据。

在实际工作中，选择合适的淘汰策略通常取决于效率和数据合理性的平衡。常用的策略是allkeys-random。

  • 如果需要确保不丢失任何数据，可以选择no-eviction策略，但需要小心内存溢出。
  • 如果希望更好地利用内存并且可以容忍一些数据被淘汰，可以选择volatile-lru或lfu策略。
  • 如果希望淘汰数据更具随机性，可以选择random策略。

最终的选择应该根据你的应用场景来进行权衡和调整。
